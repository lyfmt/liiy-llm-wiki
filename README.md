# liiy-llm-wiki

`liiy-llm-wiki` 是一个面向长期演进的本地优先知识代理项目。它想做的不是“再包一层聊天界面”，而是把资料接入、知识拆分、检索、维护、展示和学习放进同一个可持续演进的系统里。

## 当前状态

**这个项目目前仍然处于未完成状态。**

当前仓库更接近一个已经跑起来、但仍在持续搭骨架的实验系统，而不是一个可以直接交付给终端用户的成品。它已经有了后端运行时、最小 Web 界面、知识页存储、基础检索与治理机制，但距离真正稳定、完整、可持续维护的知识代理产品还有明显距离。

如果你现在阅读这个仓库，应该把它理解为：

- 一个正在演进中的知识系统原型；
- 一个用于验证长期方向的实现基础；
- 一个尚未完成的 LLM Wiki Agent 实验项目。

## 这个项目想走向哪里

这个项目的目标不是让模型“记住更多”，而是让模型始终能回到**第一手资料**。

未来理想中的系统应当是这样的：

1. 用户上传资料，或让 LLM 通过受控方式获取资料。
2. 系统不把整份资料粗暴塞给模型，而是先把资料按可维护粒度拆开。
3. LLM 在回答、维护或写回之前，先通过检索、查找、导航去拿到具体的原始知识片段。
4. 有长期价值的结果再被整理回 wiki，成为后续可继续维护和学习的知识资产。

换句话说，这个项目想做的是：

- 让资料变成结构化、可追踪、可复用的知识对象；
- 让 LLM 优先使用可定位、可回溯的原始证据；
- 让知识可以被持续维护、展示、复查和学习，而不是停留在一次性对话里。

## 未来的核心知识流转

这是目前最重要、也最值得继续做深的一条主线。

### 1. 资料接入

未来的资料来源不应只有一种，而应至少包括两类：

- 用户主动上传的资料；
- 由系统或 LLM 在受控条件下获取到的资料。

这些资料进入系统后，首先应该被登记为明确的来源对象，而不是立刻变成一段无法追踪的上下文文本。

### 2. 资料粒度拆分

未来系统不应把“一个文件”当成最小知识单位，而应继续往下拆分，例如：

- 资料级：一本书、一篇文章、一份讲义、一段网页；
- 章节级：目录、章节、子章节、表格、附录；
- 证据级：具体段落、摘录、定义、事实描述、原话片段。

只有把资料拆到足够细的粒度，后续的检索、引用、维护和学习才会可靠。

### 3. 第一手知识获取

未来 LLM 不应该优先依赖自己的参数记忆或仓库里的二手总结，而应优先通过以下方式拿到第一手知识：

- 检索相关来源；
- 查找相关章节或片段；
- 读取原始证据；
- 沿着页面、来源、标签、引用关系继续导航。

这意味着系统真正重要的不是“生成能力”本身，而是**先找到原始知识，再让生成建立在原始知识之上**。

### 4. 维护、展示与学习

未来知识进入系统后，不只是供一次回答使用，而是应该继续支持：

- 维护：更新、修正、补充、治理；
- 展示：以 wiki、来源页、主题页、检索结果页等形式呈现；
- 学习：让 LLM 在后续任务中继续复用这些已经沉淀下来的知识结构。

最终理想状态下，系统里的知识应当既能被人浏览，也能被 LLM 持续读取、维护和学习。

## 当前已经具备的基础

尽管项目还没完成，但已经有一些关键基础：

- `raw/`、`wiki/`、`schema/`、`state/` 四层目录结构；
- TypeScript 后端运行时与 CLI；
- 最小 Web 界面；
- 基础的 ingest、query、lint 与 review gate 机制；
- 可持久化的 wiki 页面与运行状态。

这些能力说明项目已经不是纯文档设想，但也还远不是完整产品。

## 当前还没有完成的部分

下面这些方向仍然明显处于未完成状态：

- 更细粒度的资料拆分与证据建模；
- 面向第一手资料的稳定检索链路；
- 更完整的来源页、主题页和导航结构；
- 更成熟的知识维护工作流；
- 更适合长期使用的前端展示层；
- 让 LLM 持续复用、持续学习沉淀知识的闭环。

## 仓库结构

- `src/`：后端、CLI、运行时、存储与知识维护逻辑
- `web/`：前端界面
- `raw/`：原始资料输入层
- `wiki/`：长期知识层
- `schema/`：规则、约束与治理策略
- `state/`：运行态、审计与中间过程
- `docs/`：规格、问题记录与设计文档

## 快速开始

```bash
npm install
npm --prefix web install
npm run lint
npm run test
npm run build
npm run build:web
```
