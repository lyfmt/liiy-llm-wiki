# Knowledge Insert 插入链路现状与后续方向报告

## 1. 文档定位


本文关注的是整条链路，而不是某一个工具或某一个测试。

---

## 2. 这条插入链路到底在做什么

从目标上讲，这条链路要解决的问题很简单：

把一份新的资料，从“上传的原文件”，变成“能长期留在 wiki 里、以后还能继续查询和扩写的知识”。

这件事并不是直接让模型看完资料后写一页总结，而是分成了多个层次：

- `source`：原始资料本身，用来保留来源身份和追溯信息。
- `section`：从原始资料中整理出来的正文单元，理想状态下应当尽量贴近原文。
- `topic`：承载多个 section 的阅读入口和总览页。
- `taxonomy`：组织 topic 的导航树。
- `graph`：把 entity、concept、section、source、evidence 等关系连起来，服务后续检索、关联和回源。

也就是说，真正承载知识正文的应该是 `section`，`topic` 负责聚合与导航
---

## 3. 当前插入链路的实际流程

当前仓库里的知识插入流程，大致可以分成下面 10 步。

### 3.1 资料进入系统

用户上传一份文档后，系统先把它登记成一个正式的 `source`。

这一步的意义是先给资料建立身份，让系统知道：

- 这份资料是什么
- 它来自哪里
- 以后如何追溯它

### 3.2 资料被整理成可处理资源

原始资料会先被转换成一份结构化资源，然后被切成很多小块。

这里的小块不是为了给人直接阅读，而是为了让 agent 能稳定处理长文、整本书、大 PDF，而不会一次吞掉太多内容。

### 3.3 长文会被分批处理

如果资料很长，系统不会交给一个 agent 一次看完，而是切成多个批次。

这样做的目的是提升稳定性，降低单次调用失败时的损失。

### 3.4 agent 对每一批做知识抽取

在每个批次里，系统会抽出几类中间结果：

- evidence 或 source anchors
- assertion
- entity / concept
- section 候选
- topic 相关提示

这一层的核心不是写成漂亮摘要，而是先把“原文里有什么”抓出来，并保留回到原文的抓手。

### 3.5 多批次结果被合并

当多个批次都处理完后，系统会把这些中间结果合并成一份统一的知识底稿。

这一步的目的是把整份资料重新看成一个整体，而不是永远停留在碎片状态。

### 3.6 section 被归并成正文单元

接下来，系统会把多个候选 section 归并成最终的 `section`。

按原始设计，这里应当得到的是一组贴源正文单元，也就是“可以代表原文某一段局部内容”的 section。

### 3.7 section 选择 topic 宿主

有了 section 之后，系统再决定这些 section 应该挂到哪个 `topic` 下。

这里的真实宿主关系应当是：

`section -> topic`

而不是：

`section -> taxonomy -> topic`

taxonomy 是 topic 的分类骨架，不是 section 的直接宿主。

### 3.8 topic 再进入 taxonomy

当 topic 决定好了之后，系统再把这些 topic 放进 taxonomy 树下，让未来的浏览和检索更自然。

### 3.9 通过门禁检查

写回之前，系统会做一些审查：

- 有没有明显漏读
- section 有没有拿到宿主
- topic 有没有进入 taxonomy
- 是否具备继续写入 wiki 和 graph 的条件

### 3.10 生成 wiki draft，并写回 graph / wiki

最后，系统会生成最终 draft。

理想情况下，写回后的 topic 页面应该保留 `topic -> sections[]` 结构，让 topic 成为总览入口，section 成为正文承载层。同时，graph 中也应该写入对应的结构关系，供后续查询和连接使用。

## 4. 当前已经做到什么

到目前为止，这条链路已经不是概念设计，而是已经有了真实落地的前半段和中段能力。

当前已经能确认做到的事情包括：

- 新资料可以进入系统并被登记成正式 source
- source 可以被整理和分块
- 长文档可以被切成多个 worker 批次
- 多个批次的抽取结果可以被合并
- section 不再只是一个空摘要壳，而开始承载正文
- topic 规划、taxonomy 规划和 topic draft 生成已经都具备基础实现
- graph 和 wiki 的写回工具链已经基本成型

换句话说，现在的问题不是“有没有链路”，而是“链路是否已经收口到足够稳定、足够合理、足够符合原始设计”。

---

## 5. 当前最核心的问题

### 5.1 长链路稳定性还不够

面对整本书或大 PDF，链路很容易在中间某个批次失败，或者在远端调用时不稳定。

这意味着系统虽然能产出很多中间结果，但完整闭环经常会卡在中后段。

### 5.2 section 的角色发生了偏移

从设计上说，`section` 应该更接近“贴源片段”。

也就是说，它应该表示的是原文中的一个局部知识单元，带有较强的来源感、边界感和回源能力。

但当前实现中的部分 section，已经越来越像“带一点正文的主题块”。

一旦 section 不再像原文片段，而更像一个语义主题块，那么它就会自然地开始接近 topic。

### 5.3 topic 被切得过细

当前已经能观察到大量“一条 section 对一个 topic”的情况。

这说明 topic 没有很好承担“聚合多个 section”的职责，反而在不少地方退化成了 section 的别名。

这种结构会带来两个后果：

- topic 页面变得很多，但每一页信息都很薄
- section 和 topic 的职责边界变得模糊

### 5.4 hint 在实际决策中的权重过高

原本 hint 只是辅助信息，不应该直接决定最终的宿主结构。

但当前链路里，topic 相关 hint 已经在很大程度上主导了 section 的归宿。

这会让系统更容易沿着“语义标签”去建 topic，而不是沿着“贴源正文归并”去找宿主。

### 5.5 taxonomy 现在更像“能生成”，还不像“能长期用”

taxonomy 这层已经有产物了，但它还没有完全达到稳定、合理、可长期维护的状态。

它现在更像是具备了骨架能力，但分类质量还没有完全收敛。

### 5.6 门禁更多是在检查“形式正确”，还没有足够检查“结构合理”

当前门禁已经能检查一些基础问题，比如是否缺少宿主、是否缺少放置位置。

但它还不太会检查下面这些更重要的问题：

- section 是否还保持贴源属性
- topic 是否切得过碎
- taxonomy 是否只是形式上挂上去了
- 写回后的页面是否真的保住了 section 粒度

---

## 6. 当前问题的本质判断

如果把问题压缩成一句话，那么当前最关键的偏移是：

系统本来应该让 `section` 成为贴源正文单元，但在实际链路中，`section` 被逐渐语义化、主题化了。

一旦 `section` 先被做成“主题块”，那么后面就会自然出现下面的滑坡：

- section 越来越像小主题
- topic 越来越像 section 的别名
- taxonomy 越来越像给这些小主题找挂载位置

这样一来，整条链路虽然还能跑，但它会逐渐偏离最初的设计目标。
