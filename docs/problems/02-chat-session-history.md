# 问题表单 02：AI Chat Page 缺少历史会话与新建会话能力

## 1. 问题标题
AI Chat 页面当前只有单一对话流，缺少历史会话列表与新建会话能力

## 2. 当前现象
- 当前页面更像“当前 run 展示器 + run history”。
- 用户可以看到已有 run 结果，但还没有清晰的“conversation/session”层。
- 用户不能自然地：
  - 创建一个新对话；
  - 在多个历史对话之间切换；
  - 保持每个对话自己的上下文。

## 3. 期望行为
- Chat 页面支持“会话”这一层级，而不只是 run。
- 用户可以：
  - 新建会话；
  - 查看历史会话；
  - 进入某个历史会话继续提问；
  - 保留每个会话自己的消息/运行记录。

## 4. 用户价值
- 更接近用户对聊天产品的基本预期。
- 把“单次运行记录”提升为“长期对话工作空间”。
- 为后续 agentic、多轮协作、上下文累积铺路。

## 5. 已知根因/背景
- 当前系统主要有 run/history，但 session/conversation 产品概念不完整。
- 已开始查看：
  - `src/domain/agent-session.ts`
  - `web/src/features/ai-chat/pages/ai-chat-page.tsx`
- 说明仓库里可能已有部分 session 基础，但前后端尚未形成完整产品闭环。

## 6. 边界与约束
- 应优先复用已有 run 存储和读取能力。
- 不要一上来就做复杂 IM 架构。
- 需要明确“session”和“run”的关系：
  - 一个 session 包含多个 run？
  - 一个 user message 触发一个 run？
- 需要考虑刷新后恢复、侧栏展示、默认选中逻辑。

## 7. 非目标
- 不是这一步就做完整 Slack/Discord 式实时协同。
- 不是这一步就做跨设备同步策略设计。
- 不是这一步就做无限复杂的上下文裁剪/摘要压缩系统。

## 8. 影响范围
- chat 页面左侧历史区
- session / conversation 数据模型
- run 与 session 关联关系
- 新建会话入口与默认 UX

## 9. 初步验收标准
- [ ] 用户可以显式新建一个新会话
- [ ] 左侧可以看到历史会话列表
- [ ] 进入历史会话后能看到对应消息/运行记录
- [ ] 在历史会话中继续发送消息时，记录进入正确会话
- [ ] 页面刷新后仍能恢复最近或指定会话
- [ ] 现有 run history 不丢失，且迁移关系清楚

## 10. 待确认点
- 会话标题是否自动从首条消息生成？
- 历史列表展示的是“会话”还是“最近运行”？
- 旧的 run 数据如何归入 session：自动迁移、默认单 run 一 session，还是仅对新数据生效？
